## 结论与边界

当前仓库没有可核对的前端项目、接口定义或业务规则文档，因此本目录只提供测试夹具和场景模板，**不声明这些字段就是现有接口契约**。接入真实项目时，必须通过适配层把真实响应转换成测试模型，并补齐 `unresolvedRules` 中的规则。

已整理的数据覆盖：正常、重复、超上限、缺字段、无库存、成功、空数组、部分字段缺失、500 错误和超时。

## 尚未确认的业务规则

以下内容不得从本测试夹具反推为产品规则：

- 最大对比数量
- 重复商品的判定依据
- 参数差异的计算与高亮规则
- 价格、库存缺失时的展示文案
- 局部字段缺失时的页面降级方式
- 商品详情跳转字段、路由名称和参数来源
- 成功码、错误码、请求标识及真实响应字段名

这些项目统一记录在 `compare-types.ts` 的 `unresolvedRules` 中。测试场景需要数值时，通过 `scenarioConfig` 显式传入仅用于测试的数据，不能把它当成生产默认值。

## 测试模型

`ProductCompareFixture` 是测试模型，不是接口 DTO：

- `fixtureId`：夹具内部标识，仅用于关联测试数据
- `source`：未解释的原始字段容器，等待真实接口映射
- `specs`：测试用规格键值；不预先写入差异结论
- `expected`：场景期望元数据，不属于接口返回
- `missingFields`：故意缺失的候选字段
- `detailCandidate`：详情跳转所需候选值；字段来源仍待确认

`CompareApiSample` 同样是传输样本容器。`payload` 保留原始响应形态，待真实接口明确后再建立强类型适配器。

## 场景说明

- 正常数据：`normalProducts`
- 重复候选：`duplicateAttempt`，判重策略为待确认项
- 超限候选：`overLimitAttempt`，上限由场景参数传入
- 缺字段：`missingFieldProduct`
- 无库存：`outOfStockProduct`
- 成功、空数组、部分字段缺失、500、超时：见 `compare-api-samples.ts`
- 页面及联调检查清单：见 `compare-scenarios.ts`

价格缺失和无库存使用两个不同的输入夹具，避免用单一输入断言两个不相符的状态。

## 接入真实项目的完成条件

在宣称“与现有接口结构一致”前，至少需要：

1. 提供真实接口定义、OpenAPI 文档或现有请求代码。
2. 确认商品唯一标识与判重规则。
3. 确认数量上限及其来源。
4. 确认差异高亮由接口提供还是前端计算。
5. 确认价格、库存兜底文案。
6. 确认详情跳转字段与路由规则。
7. 增加真实 DTO 到测试模型的适配器及契约测试。
